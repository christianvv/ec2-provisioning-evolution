# Legacy: salt-cloud EC2 Provisioning (circa 2016–2018)

This directory is a recreation of a SaltStack salt-cloud-based AWS EC2
provisioning system originally architected and implemented for a private-subnet
DevOps tooling environment on AWS. The original system was built circa 2016–2018
using SaltStack's salt-cloud module, targeting CentOS 7 as the base OS.

This is a demonstration of the architecture and design, rebuilt generically from
memory and design knowledge. It is not a copy of production code, and all
configuration values are illustrative placeholders.

---

## Context and era

**Tool:** salt-cloud, part of the SaltStack project (pre-Broadcom acquisition era).
At the time, salt-cloud was the natural provisioning extension for shops already
running Salt for configuration management — it used the same provider/minion model
and lived on the Salt master, keeping the toolchain homogeneous.

**Target OS:** CentOS 7. RHEL-compatible, well-supported by Salt's yum provider,
and standard in the environment at the time.

**Scope:** EC2 instance provisioning only. Networking resources — VPC, subnets,
security groups, Route 53 entries, IAM roles — were pre-provisioned and managed
separately. salt-cloud's job was to launch an instance, run the bootstrap script,
and hand off to Salt states.

---

## Architecture overview

```
salt-cloud master (inside VPC)
│
├── cloud.conf.d/cloud.conf                      ← global defaults (deploy script, logging)
├── cloud.providers.d/ec2-provider.conf          ← AWS credentials & provider settings
├── cloud.profiles.d/devops-tooling-profile.conf ← AMI, size, storage, tags
├── cloud.maps.d/devops-tooling-map.conf         ← named instances → profiles
└── bootstrap-scripts/bootstrap-minion.sh        ← executed on new instance via SSH
```

### Provisioning flow

1. **Operator command** — An operator on the salt-cloud master runs:
   ```bash
   salt-cloud -p devops_centos7_az1_large devops-gitlab-01
   ```
   or provisions the full environment from a map:
   ```bash
   salt-cloud -m /etc/salt/cloud.maps.d/devops-tooling-map.conf
   ```

2. **EC2 launch** — salt-cloud calls the EC2 API to launch the instance
   using the settings in the profile: AMI, instance type, AZ, subnet,
   security group, EBS volumes, and tags.

3. **SSH wait** — salt-cloud polls until the instance's private IP is
   reachable over SSH (using the provider's key pair). All SSH happens over
   private IPs since the salt-cloud master is inside the same VPC.

4. **Bootstrap script** — salt-cloud renders and executes `bootstrap-minion.sh`
   on the new instance. The script:
   - Updates the system via `yum`
   - Configures an internal yum repository (hosted in S3) and installs `salt-minion`
   - Writes the minion's pre-generated RSA key pair to `/etc/salt/pki/minion/`
     using salt-cloud template variables (`{{ vm['priv_key'] }}`, `{{ vm['pub_key'] }}`)
   - Writes the rendered minion configuration (master IP, minion ID, grains)
     to `/etc/salt/minion.d/minion.conf` via `{{ minion }}`
   - Enables and starts the `salt-minion` systemd service

5. **Minion registration** — The minion starts and connects to the Salt master.
   Because the key pair was pre-generated by the master and injected during
   bootstrap, the minion's key is already accepted — no manual `salt-key -a`
   step required.

6. **Salt states take over** — Once the minion checks in, the Salt master applies
   the appropriate states (common packages, hostname, users, services, etc.).
   State management is handled by a separate SaltStack configuration repo and is
   out of scope here.

## File structure

### `cloud.conf.d/cloud.conf`

Global salt-cloud configuration applied across all providers, profiles, and maps.
The most significant setting here is `script: bootstrap-minion`, which tells
salt-cloud which deploy script to render and execute on each new instance. Without
this, salt-cloud would fall back to the built-in generic bootstrap, which installs
Salt from the public SaltStack repo rather than from the internal yum repository.

### `cloud.providers.d/ec2-provider.conf`

Defines how salt-cloud authenticates with AWS and what defaults apply to all
provisioned instances.

Key design decisions:
- **Instance role credentials** (`use-instance-role-credentials`) — the salt-cloud
  master runs on an EC2 instance with an IAM instance profile, so no static access
  keys are stored on disk.
- **`ssh_interface: private_ips`** — salt-cloud connects to instances via their
  private IP. Since the master is inside the same VPC, public IPs were unnecessary
  and not assigned.
- **`owner` restriction** — limits AMI queries to the organization's own AWS
  account, preventing salt-cloud from accidentally pulling public marketplace AMIs
  during `--list-images`.
- **Provider-level grains** — a `cloud: devops-vpc` grain is set at the provider
  level so all instances launched through this provider are immediately targetable
  by grain in Salt.

### `cloud.profiles.d/devops-tooling-profile.conf`

Defines the instance profiles consumed by operators and maps.

Key design decisions:
- **Two base profiles, one per AZ** — `devops_centos7_az1` (us-east-1b) and
  `devops_centos7_az2` (us-east-1c). The subnet ID is the only meaningful
  difference; all other settings are identical. AZ selection was an operator
  choice at launch time rather than something the map enforced.
- **`extends` size matrix** — each base profile is extended into a full set of
  size variants (t2.micro through r4.2xlarge) using salt-cloud's `extends` key.
  This keeps each size variant to a single line and avoids duplicating the full
  profile block for every size.
- **Inline `network_interfaces`** — rather than the legacy top-level `subnetid`
  and `securitygroupid` keys, the profile uses the `network_interfaces` block,
  giving explicit control over device index and private IP assignment behavior.
- **`del_root_vol_on_destroy` / `del_all_vol_on_destroy`** — both True, so
  destroying an instance also removes all EBS volumes and prevents orphaned
  volume accumulation.
- **`rename_on_destroy`** — renames the EC2 instance (appending a timestamp
  suffix) instead of terminating immediately, providing a short safety window
  to confirm the correct instance is being targeted before permanent deletion.
- **`sync_after_install: all`** — after bootstrap completes and the minion checks
  in, salt-cloud syncs all Salt modules, grains, and states to the new minion,
  ensuring it is immediately ready for state application without a separate sync
  step.

### `cloud.maps.d/devops-tooling-map.conf`

Maps named minion IDs to profiles. A map allows the full environment to be
described declaratively and provisioned in a single `salt-cloud -m` command.
Each key is a profile name; each value is a list of minion IDs (which become
both the EC2 instance Name tag and the Salt minion ID).

The DevOps tooling environment consisted of a small set of purpose-built servers:
source control, artifact repository, directory services, CI/CD automation,
monitoring, and Salt infrastructure.

### `bootstrap-scripts/bootstrap-minion.sh`

The deploy script executed by salt-cloud over SSH on the newly launched instance.
It is referenced by name in the global cloud configuration (`script:` key in
`cloud.conf.d/cloud.conf`), and salt-cloud renders it as a Jinja template before
execution, substituting instance-specific values for the `{{ vm['priv_key'] }}`,
`{{ vm['pub_key'] }}`, and `{{ minion }}` template variables.

The internal yum repository was hosted in an S3 bucket accessible from within the
VPC, containing a curated and GPG-signed set of Salt packages. Using an internal
repo rather than the public SaltStack repo ensured version control over the Salt
release deployed across the environment, and was consistent with the security
posture of the VPC (no direct internet egress from most subnets).

## Common operational commands

```bash
# List configured providers
salt-cloud --list-providers

# List available profiles under a provider
salt-cloud --list-profiles devops_vpc

# Provision a single instance
salt-cloud -p devops_centos7_az1_large devops-gitlab-01

# Provision the full environment from the map
salt-cloud -m /etc/salt/cloud.maps.d/devops-tooling-map.conf

# Dry run — show what the map would provision without launching anything
salt-cloud -m /etc/salt/cloud.maps.d/devops-tooling-map.conf --query

# Destroy an instance
salt-cloud -d devops-gitlab-01
```

## What this system did not manage

By design, salt-cloud only provisioned EC2 instances. The following were
pre-provisioned or managed by separate tooling and are not represented here:

- VPC, subnets, route tables, internet/NAT gateways
- Security groups and network ACLs
- IAM roles and instance profiles
- Route 53 DNS records (updated manually after instance launch)
- EBS snapshots and AMI creation
- Salt state configuration (a separate SaltStack repo)
